### G1 — Phase 7 unverified

**Impact:** LEAD-01 and LEAD-02 lack independent VERIFICATION.md. Plan 07-04 (human-verify in browser) was DEFERRED at user direction ("Давай просто дальше по плану" — recorded in 07-04-SUMMARY.md). **To close:** Add a retroactive verification phase (e.g. `07.1-verify-leaderboard`) following the same pattern as Phase 5.1 / 5.2: produce `.planning/phases/07-leaderboard/07-VERIFICATION.md` via gsd-verifier goal-backward methodology, document the deferred-human-verify status, and synchronize REQUIREMENTS.md traceability. Combine with G2 fix in the same gap-closure phase.

### G2 — Leaderboard not updated on quest complete (functional bug)

**Impact:** `routers/quests.py::complete_quest` (the live JWT-authenticated quest-completion handler) does not call `leaderboard.update(redis_client, user)`. The only `leaderboard.update` call in the codebase lives inside `crud.add_reward` (legacy Telegram-id path), which is unreachable from current routers.

After server startup, `seed_if_empty` rebuilds the Redis ZSET `leaderboard:global` from DB state. Between restarts, every quest completion increments DB `lvl/xp` but leaves the ZSET stale. Users see incorrect ranks on `/app/leaderboard`, and the "you" highlight (LEAD-02) points at a stale position.

**Reproduction:**
1. Note current rank on `/app/leaderboard/me`
2. Complete a quest that grants a level up
3. Refresh `/app/leaderboard` — rank does not update until server restart

**To close:** In `routers/quests.py::complete_quest`:
- Add `redis_client: aioredis.Redis = Depends(cache.get_redis)` to the endpoint signature
- After `await db.commit()`, call `await leaderboard.update(redis_client, user)`
- Wrap in try/except matching the pattern at `crud.py:73-78` so a Redis outage does not break quest completion
- Add a unit test asserting `leaderboard.update` is invoked after a successful complete_quest

This is a small, isolated change — likely <30 lines including the test.

### G3 — AUTH-05 stub (acknowledged from prior audit)

**Status:** unchanged from 2026-04-22 audit. Email verification token is generated and validated by code, but tokens are logged to server console instead of sent via SMTP. No password-reset endpoints. Phase 11/PROD-03 will deliver `aiosmtplib` + Yandex 360 SMTP transport.

**Production-blocking?** Yes for public launch. No for internal/beta testing (manual token paste works).
